# План реализации — Telegram Mini App CalendarBot

## 1. Общий подход
1. Сначала безопасный рефактор backend для переиспользования логики.
2. Затем WebApp API.
3. Затем frontend mini app.
4. Затем интеграционное тестирование и поэтапный rollout.

## 2. Этапы реализации

## Этап M1. Контракты и флаги
1. Подготовить и утвердить контракты API mini app.
2. Зафиксировать feature flags:
1. `MINI_APP_ENABLED`
2. `MINI_APP_ADMIN_ENABLED`
3. `MINI_APP_ONBOARDING_ENABLED`
3. Зафиксировать модель RBAC (`User`, `Admin`).

### Критерии готовности
1. Контракты API согласованы.
2. Feature flags описаны в документации.
3. Нет изменений в поведении production-бота.

### Проверки
1. Документная сверка маршрутов и статусов ответов.
2. Проверка обратной совместимости текущих endpoint.

---

## Этап M2. Рефактор backend в общий service-layer
1. Вынести use-case логику из `src/telegram/bot.ts` в `src/application/*`.
2. Подключить bot-обработчики к новому service-layer.
3. Сохранить текущие доменные переходы и правила.

### Критерии готовности
1. Bot использует общий service-layer.
2. Поведение бота не изменилось.

### Проверки
1. `npm run build`
2. `npm run stage5:verify`
3. `npm run stage6:verify`
4. `npm run stage7:verify`
5. Ручной smoke `/start`, создание заявки, подтверждение, перенос, отмена.

---

## Этап M3. WebApp auth и middleware
1. Добавить валидацию Telegram WebApp `initData` на сервере.
2. Добавить механизм сессии WebApp.
3. Добавить middleware ролей для admin API.

### Критерии готовности
1. Авторизация mini app работает только через валидный `initData`.
2. Неавторизованные запросы блокируются.
3. Admin API закрыты RBAC.

### Проверки
1. Позитивный сценарий auth.
2. Негативные сценарии: подпись неверна, просроченный payload, попытка user-доступа к admin.

---

## Этап M4. User API + User UI
1. Реализовать API:
1. bootstrap
2. слоты
3. черновик
4. submit
5. мои заявки
6. отмена/перенос
2. Реализовать экраны:
1. onboarding
2. главная
3. новая заявка
4. мои заявки
5. карточка заявки
3. Применить дизайн `Neon Dark Core`.

### Критерии готовности
1. Все user-сценарии работают в mini app.
2. Отображение корректно на мобильном экране.
3. Доступны CTA и безопасные действия.

### Проверки
1. E2E: от создания заявки до просмотра статуса.
2. E2E: перенос и отмена.
3. Проверка контрастности и читаемости.

---

## Этап M5. Admin API + Admin UI
1. Реализовать API:
1. список всех заявок + фильтры
2. approve/reject
3. cancel/reschedule чужих заявок
4. чтение и изменение настроек слотов
2. Реализовать экраны:
1. admin список заявок
2. admin настройки

### Критерии готовности
1. Admin-сценарии закрыты полностью.
2. Комментарий reject опциональный.
3. Все действия проходят через общий service-layer.

### Проверки
1. E2E: approve/reject.
2. E2E: admin cancel/reschedule.
3. E2E: изменение настроек слотов и влияние на доступные слоты.

---

## Этап M6. Интеграционное тестирование двух каналов
1. Проверить согласованность bot/webapp на одной БД.
2. Проверить гонки и повторные действия.
3. Проверить журналирование и трассировку.

### Критерии готовности
1. Нет конфликтов состояния между каналами.
2. Нет дублей критичных действий.
3. Логи полные и читаемые.

### Проверки
1. User action в bot, завершение в webapp и наоборот.
2. Одновременные операции по одной заявке.

---

## Этап M7. Rollout
1. Включение mini app в `dev`.
2. Smoke в ограниченном контуре.
3. Ограниченный запуск в production.
4. Постепенное расширение доступа.

### Критерии готовности
1. Production-бот стабилен.
2. Mini app стабилен в боевых сценариях.
3. Есть наблюдаемость и журнал инцидентов.

### Проверки
1. Логи сервера и бизнес-аудит.
2. Проверка отказоустойчивости и rollback через feature flags.

## 3. Риски и меры
1. Риск: регресс bot-flow при рефакторинге.
1. Мера: сначала перенос логики без изменения поведения + регресс-прогон verify-скриптов.
2. Риск: рассинхрон bot/webapp.
1. Мера: единый service-layer + серверные status-check и lock.
3. Риск: ошибки auth webapp.
1. Мера: строгая проверка `initData` и TTL.
4. Риск: рост нагрузки на SQLite.
1. Мера: rate limit, короткие транзакции, staged rollout.
