# Техническое задание — Telegram Mini App CalendarBot

## 1. Цель
Создать Telegram Mini App как дополнение к существующему production-боту CalendarBot без поломки текущих сценариев в боте.

## 2. Ключевые принципы
1. Не дублировать бизнес-логику: mini app и bot используют общий backend-use-case слой.
2. Не изменять поведение текущего Telegram-бота без явной необходимости.
3. Использовать ту же БД и те же доменные статусы/правила.
4. Все изменения внедрять поэтапно и с обратной совместимостью.
5. Разработка ведется в ветке `dev`.

## 3. Роли
1. `User`
2. `Admin` (один администратор на старте, по `telegram_id` владельца)

## 4. Функции User
1. Создание заявки через мастер.
2. Просмотр “моих заявок”.
3. Просмотр карточки заявки.
4. Отмена своей заявки.
5. Перенос своей заявки.

## 5. Функции Admin
1. Просмотр “всех заявок”.
2. Фильтры в списке заявок: статус, период по дате.
3. Подтверждение/отклонение заявок (комментарий при отклонении опциональный).
4. Перенос/отмена чужих заявок.
5. Управление настройками слотов (рабочее время, буфер, горизонт, лимит, опережение).

## 6. UX и дизайн
1. Интерфейс mobile-first (основной сценарий использования — телефон).
2. Экраны MVP:
1. `Onboarding`
2. `Главная`
3. `Новая заявка`
4. `Мои заявки`
5. `Карточка заявки`
6. `Admin: Все заявки`
7. `Admin: Настройки`
3. Стиль: `Neon Dark Core`.
4. Палитра: акценты `#00E5FF` и `#A3FF12`.
5. Анимации: быстрые и сдержанные.
6. Обязательная читаемость и высокий контраст.
7. Добавить в onboarding поддержку “Добавить на главный экран”.

## 7. Запуск и навигация
1. Mini app запускается кнопкой в боте (menu button).
2. В onboarding показать назначение mini app и быстрый вход в основные сценарии.

## 8. Безопасность
1. Серверная валидация Telegram WebApp `initData`.
2. Серверная авторизация пользователя только по данным `initData`.
3. RBAC для admin-методов.
4. Rate limit для действий создания/подтверждения/переноса/отмены.
5. Идемпотентность и/или блокировки для защиты от повторных кликов и гонок.

## 9. Консистентность между каналами
1. Bot и mini app работают параллельно с одной БД.
2. При изменении заявки в одном канале второй канал должен видеть актуальный статус.
3. Невалидные действия для текущего статуса блокируются.

## 10. Логирование
1. Сохраняется текущий JSON-формат технических логов.
2. Добавляется признак канала: `channel=bot|webapp`.
3. Сохраняется текущий бизнес-аудит в таблице `ActionLog`.

## 11. Нефункциональные требования
1. Производительность интерфейса на мобильных устройствах приоритетна.
2. Без тяжелых визуальных эффектов, ломающих плавность.
3. Совместимость с текущим production-развертыванием без остановки бота.

## 12. Критерии приемки MVP
1. User-сценарии mini app проходят end-to-end.
2. Admin-сценарии mini app проходят end-to-end.
3. Текущие bot-сценарии не регрессируют.
4. Все операции корректно логируются.
5. Mini app включается/выключается без влияния на рабочий bot-flow.


## 13. Статус реализации
1. Статус на 2026-05-06: требования ТЗ реализованы.
2. User и Admin сценарии работают в production-контуре.
3. Доработки по приемке (UX и устойчивость отмены/переноса) закрыты.
4. Регрессии существующего bot-flow не выявлены по итогам проверок.
