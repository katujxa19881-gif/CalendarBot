## 1. Итоговая рекомендация для текущего MVP
С учетом вашей цели сделать недорогой тестовый запуск, я рекомендую такой стек:

1. Backend: `Node.js`
2. Язык: `TypeScript`
3. Telegram-бот: `grammY`
4. Webhook и HTTP-слой: `Fastify`
5. База данных: `SQLite`
6. Работа с БД и миграциями: `Prisma ORM`
7. Валидация: `Zod`
8. Календарь: `Google Calendar API`
9. Авторизация Google: `OAuth 2.0`
10. Email-уведомления: `SendGrid`
11. Хостинг: `Railway`
12. Логи: `JSON logs` + бизнес-журнал в базе

---

## 2. Почему выбран именно такой вариант
1. Он дешевле, чем схема с отдельным `PostgreSQL`.
2. Он подходит под ваш ожидаемый объем: несколько заявок в день.
3. Он позволяет держать MVP в одном сервисе без лишней инфраструктуры.
4. Он удобен для реализации по этапам вместе со мной.
5. Он достаточно надежен для тестового запуска, если не усложнять архитектуру.

## 3. Подробно по технологиям

## 3.1. `Node.js + TypeScript`
### Зачем нужно
1. Один язык для всей backend-логики.
2. Удобно для Telegram webhook, Google API и email API.
3. `TypeScript` заранее отлавливает часть ошибок в данных и статусах.

### Альтернативы
1. `Python + aiogram`
2. `Node.js + JavaScript`
3. `NestJS`

### Почему не они
1. `Python + aiogram` хорош для ботов, но делит стек, если позже появится Mini App или веб-часть.
2. `JavaScript` проще на старте, но хуже защищает от ошибок в сложной логике заявок и статусов.
3. `NestJS` слишком тяжел для такого MVP.

### Итог
1. `Node.js + TypeScript` дает лучший баланс скорости и надежности.

---

## 3.2. `grammY`
### Зачем нужно
1. Удобно обрабатывает команды, кнопки и callback-события Telegram.
2. Хорошо подходит для пошагового мастера записи.
3. Хорошо работает с `TypeScript`.

### Альтернативы
1. `Telegraf`
2. Прямая работа с Telegram Bot API

### Почему не они
1. `Telegraf` тоже подходит, но для нового TypeScript-проекта `grammY` обычно чище и удобнее.
2. Прямая работа с Telegram Bot API даст больше ручной обвязки и больше риска ошибок.

### Итог
1. Для сценарного бота с заявками и согласованием `grammY` рациональнее.

---

## 3.3. `Fastify`
### Зачем нужно
1. Поднимает webhook endpoint.
2. Дает health-check и служебные HTTP endpoint.
3. Не перегружает проект.

### Альтернативы
1. `Express`
2. `NestJS`
3. Голый `http`

### Почему не они
1. `Express` допустим, но не дает здесь практического преимущества.
2. `NestJS` тяжелее, чем нужно для MVP.
3. Голый `http` создаст лишнюю ручную работу.

### Итог
1. `Fastify` достаточно простой и достаточно структурный.

---

## 3.4. `SQLite`
### Зачем нужно
1. Позволяет не поднимать отдельный сервер базы данных.
2. Снижает стоимость тестового запуска.
3. Подходит для низкой нагрузки и одного экземпляра приложения.

### Альтернативы
1. `PostgreSQL`
2. `MongoDB`
3. `SQLite` не использовать и хранить что-то в файлах или памяти

### Почему не они
1. `PostgreSQL` надежнее и лучше масштабируется, но для тестового MVP дороже и сложнее по инфраструктуре.
2. `MongoDB` не дает здесь преимущества, потому что у нас статусная и связанная доменная модель.
3. Хранение в памяти или файлах не подходит для production-подобного теста.

### Ограничения `SQLite`
1. Только один основной сервис приложения.
2. Нежелательно разносить web и worker по разным сервисам.
3. Нельзя рассчитывать на горизонтальное масштабирование.
4. База должна лежать на persistent volume.
5. При конкурирующих записях возможны блокировки уровня файла.

### Что это значит для проекта
1. Архитектура MVP будет упрощенной.
2. Все фоновые задачи будут выполняться внутри того же приложения.
3. Для вашего случая это допустимо и рационально.

---

## 3.5. `Prisma ORM`
### Зачем нужно
1. Упрощает схему данных и миграции.
2. Дает читаемую модель сущностей.
3. Работает и с `SQLite`, и с `PostgreSQL`.

### Альтернативы
1. `Drizzle ORM`
2. Прямой SQL

### Почему не они
1. `Drizzle` тоже возможен, но не дает здесь явного выигрыша.
2. Прямой SQL увеличит объем ручной работы и цену ошибок на старте.

### Итог
1. `Prisma` удобнее для поэтапной разработки MVP.

---

## 3.6. `SendGrid`
### Зачем нужно
1. Отправка подтверждений и напоминаний на email.
2. Удобный API для транзакционных писем.

### Альтернативы
1. `Mailgun`
2. SMTP
3. `Resend`

### Почему не они
1. `Mailgun` допустим, но не дает решающего преимущества.
2. SMTP потребует больше ручной настройки и хуже диагностируется.
3. `Resend` тоже возможен, но для этой задачи `SendGrid` достаточно типовой и удобный вариант.

### Итог
1. Берем `SendGrid`, но слой уведомлений проектируем так, чтобы потом можно было заменить провайдера.

---

## 3.7. `Railway`
### Зачем нужно
1. Подходит под модель “один сервис + volume”.
2. Упрощает старт тестового MVP.
3. Дает volume для хранения файла `SQLite`.

### Альтернативы
1. `Render`
2. VPS
3. `AWS/GCP/Azure`

### Почему не они
1. `Render` тоже возможен, но для варианта с `SQLite` модель `Railway` выглядит естественнее.
2. VPS даст больше ручной инфраструктурной работы.
3. Большие облака избыточны для такого старта.

### Итог
1. Для текущего варианта `SQLite` предпочтительнее `Railway`.

## 4. Архитектура MVP

## 4.1. Общая идея
Весь проект строится вокруг одного backend-сервиса. Он:
1. принимает webhook от Telegram;
2. хранит состояние в `SQLite`;
3. обращается к Google Calendar;
4. отправляет email;
5. сам выполняет фоновые задачи.

---

## 4.2. Компоненты
1. Telegram Bot Layer
   Обрабатывает команды, кнопки, сценарии мастера, подтверждения, переносы, отмены.
2. Application Layer
   Хранит бизнес-правила: статусы, проверки, слоты, ограничения, TTL, согласование.
3. Integration Layer
   Работает с Telegram API, Google OAuth, Google Calendar API и SendGrid.
4. Persistence Layer
   Хранит пользователей, заявки, черновики, календарные события, журнал действий и фоновые задачи.
5. Internal Scheduler
   Выполняет напоминания, retry и очистку TTL внутри того же сервиса.

---

## 4.3. Ограничения архитектуры
1. Один инстанс приложения.
2. Один volume для файла базы данных.
3. Без отдельного worker на MVP.
4. Без горизонтального масштабирования на старте.

---

## 4.4. Почему это нормально
1. У вас очень низкая ожидаемая нагрузка.
2. Это дает минимальную стоимость старта.
3. Это позволяет быстро получить рабочий MVP.
4. Если MVP себя покажет, потом можно перейти на `PostgreSQL`.

---

## 5. Логи и диагностика

## 5.1. Какие логи обязательны
1. Технические логи приложения.
2. Логи интеграций.
3. Бизнес-журнал действий.
4. Логи фоновых задач.

## 5.2. Что должно быть в техническом логе
1. `timestamp`
2. `level`
3. `operation`
4. `request_id`
5. `user_id` или `actor_id`
6. `entity_id`
7. `status`
8. `error_code`
9. `error_message`

## 5.3. Что должно быть в бизнес-журнале
1. Создание черновика.
2. Восстановление черновика.
3. Отправка заявки.
4. Подтверждение.
5. Отклонение.
6. Перенос.
7. Отмена.
8. Создание события в календаре.
9. Отправка уведомления.
10. Ошибка интеграции.

## 5.4. Для чего это нужно
1. Чтобы быстро понимать, где именно произошла ошибка.
2. Чтобы не разбирать проблему вслепую через код.
3. Чтобы по каждому этапу у нас была техническая диагностика.

---

## 6. Что зафиксировано на текущий момент
1. Для тестового MVP выбираем экономичную архитектуру.
2. База данных: `SQLite`.
3. Хостинг: `Railway`.
4. Один backend-сервис без отдельного worker.
5. Все этапы реализации должны учитывать обязательное логирование.
